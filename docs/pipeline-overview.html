The Artifact Pipeline — Live Architecture

The full TinyCloud producer/consumer loop is live in production. A user signs in at TinyFeed with a passkey, the front end auto-delegates their Listen-read + artifacts-read/write scopes to a TEE-hosted distillery agent, and one click generates a tweet + article that the agent publishes to the user's own xyz.tinycloud.artifacts space — which a pure-client feed then renders. Producer and consumer talk only through TinyCloud. Every box and arrow below traces to a deployed file, skill, or recorded decision.

§1Live production flow — passkey → TEE agent → the user's TinyCloud space

The end-to-end loop is deployed. A user signs in at TinyFeed (Cloudflare Pages) with an OpenKey passkey over a broadened manifest; the web-SDK session is persistent (restoreSession, 7-day TTL) and on sign-in the front end auto-delegates the Listen-read + artifacts-read/write scopes to the TEE-hosted agent's DID. One Generate click hits the agent, which reads the user's Listen transcripts, distills a tweet + article under the delegation, and publishes them via tc-publish to the user's own xyz.tinycloud.artifacts space. The feed (a pure-client web-SDK app) reads SQL and renders.

Origins (historical). The pipeline began as a local-disk Folio setup — a hono server that scanned artifacts/<type>/<slug>/artifact.json off disk behind an OpenKey gate. That is gone: the consumer is now a pure-client web-SDK app and there is no disk-scanning server in the live path. The quality machinery (extract → triage → draft → critic → verify-quotes) carried over verbatim into the agent's generate stage.

§2The TEE boundary & the delegation trust model

A deeper view of where the agent runs and under whose authority it acts. The agent is deployed to a Phala Confidential VM (Intel TDX), holds a stable did:pkh on a named volume, and only ever acts under the user's signed delegation, writing to the user's own space — never an owner key. The same per-user delegation is the real access control; the API bearer token in the client bundle is only a casual-access gate.

Security posture (live). LIVE TEE hosting, stable DID on a named volume, two-phase delegation validation, multi-origin exact-match CORS, the sandboxed-HOME delegation threading (no owner key on disk), and a scrubbed-env + tool-restricted claude -p generate step over untrusted transcript text. REMAINING Phase-2 real per-user auth (the bundled AGENT_API_TOKEN is a casual gate, not a secret — the per-user delegation is the real protection).

§3The delegation handshake — front-end OpenKey flow

How the user grants the TEE agent read/write into their own space, with no extra wallet prompt. The scopes are baked into the signed OpenKey recap at sign-in, so delegateTo is silent. This is the live front-end flow (it replaced the earlier CLI request/grant file hand-off).

What changed. The earlier design was a CLI file hand-off (auth request --emit → human auth grant → auth import → auth retry). The live flow is a single browser handshake: scopes ride the signed OpenKey recap, the front end calls delegateTo with no extra prompt, and POSTs the serialized delegation to the agent. No out-of-band file passing, no human approval step.

§4One artifact's lifecycle (live)

A single artifact from transcript to rendered card, as it runs in production: generated and published by the TEE agent under the user's delegated session key, into the user's own space, with no human approval gate (artifacts publish approved), and read by a pure-client web-SDK feed.

Publish-path notes (live). The verified KV write is a base64 string value (binary KV round-trips as {"type":"Buffer",…}, so media goes in as a base64 string); tc kv put --space and the binary round-trip shipped in the js-sdk betas (cli@0.7.0-beta.4, node-sdk@2.4.0-beta.2, web-sdk@2.4.0-beta.2) that the deployed agent uses. SQL writes into the user's space work via the delegated session.

Decisions:

  • SQL is canonical (stateful, replication-ready); KV holds media (base64 string values).
  • Feed renders off typed columns + the raw_artifact JSON per type.
  • Lives in the user's own xyz.tinycloud.artifacts space (each user publishes to their own space).
  • The agent writes only feed + media (publish-only delegation); rows land approval_status='approved'.

The owner's front-end session bootstraps feed + interactions on connect. The agent's delegation is interactions [read] only (never write) — preserving the reader-writes / agent-reads split. Closing the preference loop off these rows (distill-preferences reading SQL) is the remaining read-side work (§7).

Note. The contract supports 8 artifact types; the live run generates two per run — a tweet (banger-extractor) and an article (write-article + hero) — and the feed renders those card types. raw_artifact preserves the original type and all fields, so no contract data is lost; the SQL type column is the feed's render selector.

§7Remaining work

The loop is live; these are the genuine follow-ups that have not yet shipped or deployed.

  • Closed preference loop (read side). REMAINING The interactions table is bootstrapped and the six-action + distill-preferences machinery exists, but distill-preferences reading those SQL rows to drive [learned] bullets is not yet wired into the live loop.
  • tinycloud-node #78 not yet deployed. REMAINING The CREATE INDEX authorizer fix is merged and Codex-clean but has not been deployed to the live node.
  • Smithers React version skew. REMAINING The Smithers form of the run (.smithers/workflows/agent-run.tsx) is not yet runnable: the local .smithers orchestrator pins smithers-orchestrator ^0.20.4 while the global CLI is 0.22.0 (a React-version skew that blocks graph/run). Until they align, /agent/run runs the same stages directly via runner.ts.
  • Longer OpenKey grant TTL. REMAINING The session/delegation TTL is 7 days; a longer grant TTL is a deferred slice.
  • Phase-2 real per-user auth. REMAINING The bundled AGENT_API_TOKEN is a casual-access gate (the per-user delegation is the real protection); real per-user auth is phase-2.
  • Phase-2 agent runtime. REMAINING The backend uses claude -p, not Eliza; adopting @tinycloud/agent-client / tinycloud-agents is a deferred phase-2 step.
