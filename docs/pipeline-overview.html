§1Current state — local disk, no TinyCloud

Today the pipeline is entirely local. Transcripts come from local directories ($TRANSCRIPT_DIRS); skills do extract → triage → draft → critic → verify-quotes → save and write artifacts/<type>/<slug>/artifact.json to local disk; the Folio feed is a local hono server that scans that disk on every request. There is no TinyCloud integration in the running code.

Fidelity note. The plan sometimes calls the current feed a "Folio PWA that reads artifacts from disk." Accurate, but the running code is more than a static reader: it is an authenticated hono server (OpenKey passkey front-door, /api/* + /media/* gated) that also serves a preferences panel and a Generate button (POST /api/generate) which fires the same feed-run recipe on demand. The cards themselves are still scanned from local disk.

§2Ideal state — TinyCloud producer/consumer loop

Target: both sides talk only through TinyCloud. Listen transcripts are read via tc under a delegation the agent requests from the user (the agent holds its own key, delegate mode). Reframed skills produce tweet / article / video artifacts; a new tc-publish step writes them to a hosted xyz.tinycloud.artifacts applications space (hybrid SQL rows + KV image blobs). A new pure-client Feed web viewer renders the cards and writes interaction metadata back to SQL; distill-preferences closes the loop. The whole thing is a Smithers workflow with a human-approval gate for outward artifacts.

What's actually new vs. reused. BUILT the miner skills, the artifact contract, the critic/verify loop, distill-preferences. PLANNED tc-listen-read (with delegation request), the video skill, tc-publish, the Smithers workflow + approval gate, the pure-client Feed viewer, and the interaction-metadata write-back. IN-PROGRESS the js-sdk CLI write-path fixes (kv put --space, binary KV round-trip) and the space host verb that makes a hosted applications space reachable headlessly.

§3The delegation handshake

How a non-owner agent (its own key) gains read/write into the owner's applications space. Verified end-to-end headless between two local identities (.context/tc-write-delegation-verified.md, Q4). There is no node-side approval inbox — the request/grant artifacts are the hand-off medium.

Open delta (delegation UX). The plan wants tc-listen-read to "request a delegation from the user and wait." The verified ground truth: there is no built-in polling URL or hosted approval queue — the agent emits a request artifact, a human runs auth grant out-of-band and returns the grant JSON, then the agent imports it and polls locally with auth retry. The "wait" is a human-in-the-loop file hand-off, not a server round-trip. (PLANNED skill, BUILT primitives.)

§4One artifact's lifecycle (ideal)

A single artifact from transcript to rendered card, threading the existing quality loop through the new TinyCloud publish target and the Smithers approval gate.

Known publish-path caveat. tc kv put currently has no --space flag (writes the profile's primary space only), and binary via kv put --file round-trips as JSON {"type":"Buffer",…}. The verified workaround is base64-into-a-string-value; the proper fix (kv put --space + binary round-trip) is the IN-PROGRESS js-sdk task. SQL writes into --space applications already work once the space is hosted.

§7Where the code and the plan diverge

High-signal mismatches a reader should know about. Code is shown as current; the plan as ideal.

  • Feed is a server, not a static reader. Plan: "Folio PWA reads artifacts from disk." Code: harness/feed/src/server.ts is an authenticated hono server (OpenKey passkey, gated /api/* + /media/*) with a preferences panel and a Generate button. The ideal viewer is the opposite direction — a pure-client web-SDK app with no server.
  • Type count: 8 → 3. Code: formats.ts defines 8 artifact types. Plan: the Feed v1 renders 3 (tweet/article/video). No data loss — raw_artifact JSON keeps the full contract; the SQL type column is just the render selector.
  • "Request delegation and wait" has no server poll. Plan implies the agent waits on an approval. Verified code: the SDK uses request/grant artifact files (auth request --emit → human auth grant → auth import → local auth retry poll). The wait is a human file hand-off, not a hosted approval queue.
  • KV write into applications is blocked from the CLI today. tc kv put ignores --space (primary space only) and binary round-trips as JSON. SQL writes into --space applications work once the space is hosted; KV needs the IN-PROGRESS kv put --space fix (base64-string workaround verified meanwhile).
  • tc space create is dead. Spike finding (.context/applications-space-hosting.md): there is no tinycloud.space/create handler in the node; hosting goes through a space/host SIWE → /delegate. The owner (root authority) can host its own applications space headlessly — the prior "hosting is impossible" premise was wrong.
  • Outward comms skills aren't wired into the automated loop yet. README "routing seam (in flight)": today feed-run drives the internal miners; banger-extractor / investor-snippet / quote-card are invoked directly by an agent, and the approvals surface itself isn't built. The ideal pulls them into the Smithers workflow behind the approval gate.
